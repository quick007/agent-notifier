# Agent Notifier Protocol Foundation

Status: implementation foundation, security review required before marketing claims.

## Principles

- Message content is end-to-end encrypted before it reaches the server.
- The server may route, queue, expire, rate-limit, and deliver encrypted envelopes, but it must not read titles, bodies, reply text, approval text, sensitive flags, or approval decisions.
- Sender and device authenticity use signed canonical payloads, not long-lived bearer tokens.
- All signed payloads include `schemaVersion` and a domain separation string.
- Duplicate suppression that compares plaintext belongs in the local sender runtime only.

## Package Split

`@agent-notifier/protocol` owns deterministic wire serialization:

- `canonicalJson(value)`: sorts object keys, rejects non-JSON values, rejects non-finite numbers and `-0`, and refuses non-plain objects.
- `canonicalSigningText({ domain, payload })`: returns `domain + "\n" + canonicalJson(payload)`.
- `canonicalSigningBytes(...)`: UTF-8 bytes for ECDSA signatures.
- `messageContentAadBytes(metadata)`: canonical metadata AAD bytes for content encryption.
- `messageEnvelopeSigningBytes(envelope)`: canonical sender-signed stored-envelope bytes.
- `responseEnvelopeSigningBytes(envelope)`: canonical device-signed encrypted response bytes.
- `assertKnownCriticalFields(...)`: rejects unknown or missing fields listed in a payload's `critical` array.
- Shared v1 wire types for message metadata, encrypted envelopes, key wraps, and responses.

`@agent-notifier/crypto` owns small WebCrypto wrappers:

- P-256 ECDH key generation/import/export and shared-secret derivation.
- P-256 ECDSA SHA-256 signing and verification.
- AES-256-GCM encryption and decryption.
- HKDF-SHA-256 derivation.
- Base64url encoding and decoding.
- Content encryption, per-device key wrapping, and content opening helpers.

The crypto package intentionally accepts caller-provided AAD/signing bytes. App, CLI, MCP, and PWA code should get those bytes from the protocol package and should not invent ad hoc JSON or string signing formats.

## Domains

Initial v1 domains:

- `agent-notifier/message/v1`
- `agent-notifier/message-content-aad/v1`
- `agent-notifier/response/v1`
- `agent-notifier/pairing/v1`
- `agent-notifier/request/v1`
- `agent-notifier/key-wrap/v1`

The signed bytes are always:

```text
<domain>
<canonical-json-payload>
```

The payload must contain `schemaVersion: 1`. Unknown critical fields must be rejected before signing or accepting a signature.

## Message Envelope Shape

The server-visible metadata is signed but not encrypted. Message content is encrypted:

```json
{
  "schemaVersion": 1,
  "messageId": "msg_...",
  "metadata": {
    "schemaVersion": 1,
    "messageId": "msg_...",
    "recipientId": "rcp_...",
    "senderId": "snd_...",
    "mode": "notify",
    "createdAt": "2026-06-30T00:00:00.000Z",
    "expiresAt": "2026-07-14T00:00:00.000Z"
  },
  "ciphertext": "base64url...",
  "contentNonce": "base64url...",
  "contentAadHash": "base64url...",
  "keyWraps": [
    {
      "schemaVersion": 1,
      "deviceId": "dev_...",
      "ephemeralPublicKey": "base64url-spki...",
      "wrappedKey": "base64url...",
      "wrapNonce": "base64url..."
    }
  ],
  "senderSignature": "base64url..."
}
```

`contentAadHash` is a SHA-256 hash of canonical encrypted metadata AAD from `messageContentAadBytes(metadata)`, not a plaintext hash. It exists so clients can cheaply reject mismatched AAD before attempting content decryption.

## No Server-Visible Plaintext Hashes

Do not send unsalted or server-visible hashes of title, body, request text, reply text, approval text, or decisions.

For retries, use opaque idempotency keys generated by the sender runtime. For exact duplicate suppression, compare plaintext locally before encryption. Server rate limits should use metadata only, such as sender ID, recipient ID, IP or install fingerprint, pairing attempt counters, and aggregate counts.

## Sibling Usage

Hono/Zod/OpenAPI schemas should import protocol tuple constants such as
`MESSAGE_MODES`, `RESPONSE_KINDS`, `DELIVERY_STATES`, and `SENDER_KINDS` rather
than repeating enum literals. API request and response contract types should
prefer `MessageSubmissionEnvelope`, `PendingMessageEnvelope`, and
`ResponseSubmissionEnvelope` from `@agent-notifier/protocol`. The Hono layer can
derive runtime schemas from those constants, but it must still validate base64url
format, timestamps, IDs, and authentication headers at the API boundary.
Signed sender/device request authentication should use `canonicalRequestText` or
`canonicalRequestBytes` with `REQUEST_SUBJECT_TYPES`; it is separate from
envelope signatures and signs method, path, body SHA-256, timestamp, nonce,
subject type, and subject ID.

For response submissions, `messageId` is part of the device-signed response
payload. If the API keeps `messageId` in the Hono route path instead of the JSON
body, the route must compose that path value with the validated response body
before calling `responseEnvelopeSigningBytes(...)` and verifying
`deviceSignature`.

Sender-side send flow:

1. Build `ServerVisibleMessageMetadata` and plaintext content locally.
2. Build `contentAadHash` from `messageContentAadBytes(metadata)`.
3. Encrypt content and wrap the content key once per device with `@agent-notifier/crypto`.
4. Build sender signature bytes with `messageEnvelopeSigningBytes(envelope)`.
5. Sign canonical bytes with the sender signing private key.
6. Send only metadata, ciphertext, nonces, key wraps, AAD hash, and signature to the API.

Device-side receive flow:

1. Rebuild canonical signed bytes with `messageEnvelopeSigningBytes(envelope)`.
2. Verify the sender signature before showing or storing the message.
3. Pick this device's key wrap, unwrap the content key, and decrypt locally.
4. Store plaintext only in local device storage.

Response flow:

1. Device encrypts reply or approval plaintext to the sender encryption key.
2. Device signs canonical response bytes from `responseEnvelopeSigningBytes(envelope)`.
3. Server stores and relays only the encrypted response envelope.
